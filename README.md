# Выбор фреймворка для создания микрофронтендов

Проанализировав структуру проекта, я выделил следующие особенности:

1. Основная технология, использованная для написания фронтенда — **React (17.0.2)**.
2. Используется редирект по URL для **Log in** и **Sign up**.
3. Некоторые компоненты переиспользуются несколько раз. Например, **PopupWithForm** используется во всех компонентах с поп-апами (**AddPlacePopup, EditAvatarPopup** и т. д.).
4. В `App.js` создаются общие переменные состояния, такие как **currentUser** и **email**.
5. В приложении используется общий слой с утилитами. **API** и **auth** используются всеми компонентами.

## Вывод

Для разбиения приложения на микрофронтенды в данном случае больше подходит **Module Federation**. Он хорошо зарекомендовал себя в случаях, когда:

1. В приложении используется единый технический стек. В данном случае приложение **Mesto** использует только **React.js**.
2. Переиспользуются компоненты и модули. **Mesto** использует общий **API** и **auth** слой. Некоторые компоненты ссылаются на другие (например, **PopupWithForm**).

Также **Module Federation** позволит оптимизировать загрузку библиотек и динамически подгружать требуемые микрофронтенды, что сэкономит ресурсы и оптимизирует бандл.

### Минусы

- **Module Federation** не очень хорошо оптимизирован для маршрутизации и разделения приложения на полностью независимые микрофронтенды. Однако в **Mesto** нет сложной маршрутизации, а полной независимости микрофронтендов не требуется.
- При необходимости маршрутизацию можно реализовать при помощи **SSR**.

Исходя из вышеперечисленного, наиболее подходящим фреймворком для приложения **Mesto** является **Module Federation**.

# Разделение на микрофронтенды

Исходный монолит разбит на микрофронтенды по принципу **вертикальной нарезки**. Каждый микрофронтенд отвечает за свою отдельную бизнес-функцию. Например:

- **auth-microfrontend** отвечает за логин и регистрацию пользователя.
- **profile-microfrontend** отвечает за работу с профилем пользователя.

Компоненты содержат необходимые утилиты и стили для независимой работы. Общие стили веб-страницы находятся в **host**, который отвечает за общее отображение контента, секций и страниц.

В отдельный микрофронтенд вынесен функционал работы с "картами" и тултипами, так как они унифицированы по приложению и могут работать независимо.

Так как приложение обладает сложным пользовательским интерфейсом, обслуживающим разные бизнес-функции, данный подход является наиболее подходящим.

## Структура приложения

```plaintext
/host-microfrontend                 // Микрофронтенд, который хостит и оркестрирует остальные
    /src
        /components
            App.js                  // Основной компонент, в который импортируются все микрофронтенды
            Footer.js               // Компонент футера
            Header.js               // Компонент хэдера
            Main.js                 // Основной компонент
            ProtectedRoute.js       // Компонент маршрутизации
        /styles
            /page                   // Папка со стилями для страницы
            /content                // Папка со стилями контента
        /utils
            api.js                  // Утилиты для API
        /contexts
            CurrentUserContext.js    // Контекст пользователя
        index.js                    // Точка входа микрофронтенда
    package.json                    // Зависимости и скрипты микрофронтенда
    webpack.config.js

/auth-microfrontend
    /src
        /components
            Login.js                // Компонент входа
            Register.js             // Компонент регистрации
        /styles
            login.css               // Стили для входа
            /auth-form              // Стили для формы аутентификации
        /utils
            auth.js                 // Утилиты для аутентификации
        index.js                    // Точка входа микрофронтенда
    package.json                    // Зависимости и скрипты
    webpack.config.js

/profile-microfrontend
    /src
        /components
            EditAvatarPopup.js      // Компонент изменения аватара
            EditProfilePopup.js     // Компонент изменения профиля
            PopupWithForm.js        // Универсальный поп-ап
        /styles
            /profile                // Стили профиля
            /popups                 // Стили поп-апов
        /utils
            api.js                  // Утилиты API
        index.js                    // Точка входа
    package.json                    // Зависимости и скрипты
    webpack.config.js

/card-microfrontend
    /src
        /components
            Card.js                 // Компонент карт
            ImagePopup.js           // Компонент изображения карты
            PopupWithForm.js        // Поп-ап
        /styles
            /card                   // Стили карт
            /popup                  // Стили поп-апов
        /utils
            api.js                  // Утилиты API
        index.js                    // Точка входа
    package.json                    // Зависимости и скрипты
    webpack.config.js

/info-microfrontend
    /src
        /components
            InfoTooltip.js          // Тултип
        /styles
            /popup                  // Стили тултипа (используют классы поп-апов)
        index.js                    // Точка входа
    package.json                    // Зависимости и скрипты
    webpack.config.js

/place-microfrontend
    /src
        /components
            AddPlacePopup.js        // Компонент добавления места
            PopupWithForm.js        // Универсальный поп-ап
        /styles
            /places                 // Стили компонента мест
            /popups                 // Стили поп-апов
        index.js                    // Точка входа
    package.json                    // Зависимости и скрипты
    webpack.config.js
```

Этот подход позволяет эффективно управлять зависимостями, переиспользовать компоненты и обеспечивать гибкость при разработке новых функций.
