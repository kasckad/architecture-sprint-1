# Уровень 1. Проектирование

Чтобы наиболее точно подобрать фреймворк для микрофронтенд архитектуры, необходимо определить требования
приложения и учесть особенности самих фреймворков.

Допустим аудитория приложения постоянно растёт, мы получаем переток аудитории из одной заблокированной
зарубежной соцсети. Раньше пользователям нравился минимализм приложения, но сейчас есть потребность в
быстрой импелминтации новых фич, чтобы удержать приходящую аудиторию. В дальнейшем, необходимо продумать
систему монетизации, для обеспечения стабильного бюджета на разработку.

Приложение было написано небольшой, но опытной командой - двумя сеньйор разработчиками. Их стек это _**React**_.
Логично будет развивать проект в ближайшей перспективе на этом же стеке. Так как нанимать новых людей
будет проще со одинаковым стеком. 

UI ещё не перегружен функционалом, но планируется несколько крупных фич. Независимость развёртвывания необходима,
чтобы избежать конфликтов при быстрой разработке нескольких независимых фич в параллель. Например,
решаем разрабатывать личный кабинет с фото/видео редакторами и дашборды аналитики по охвату аудитории
(геолокации подписчиков, конверсии просмотров -> лайков в зависимости от даты публикации поста, и тд.)
Из-за объемности фич, возможно, приложение столкнется с проблемами перфоманса, одним из решением может быть
lazy-loading компонентов.

С предложенными исходными данными, логичнее всего выбрать **Module Federation**.
Текущее приложение можно разбить на следующие независимые модули

    host
    profile
        -авторизация
        -редактирование профиля
    cards
        -плитка главной страницы
        -карточка
        -форма добавления контента

Таким образом добавление нового функционала возможно без вмешательства в уже работающий функционал.
Для простых запросов на обновление профиля / контента будет использоваться **HTTP API**,
с лонг поллингом для процессов обработки файлов на стороне бэкенда и отображения статуса пользователю. Глобальный стейт
будет тоже присутствовать для контекста авторизации пользователя, уведомлений из разных модулей о завершении какого-либо
процесса.

Сейчас приложение взаимодействует с одним бэкенд сервисом. Очевидно, что при добавлении фич по аналитике и
редактированию контента придётся добавлять много функционала на бэк. Смешивать данные функциональности нет необходимости,
они имеют совершенно разный стек технологий и бизнес задачи. Выбираем реализацию **BFF** - нивелируем возможность аффекта 
новых нестабильных фич на всё приложение. Маршрутизация всез запросов будет осуществляться **gateway** сервисом,
после запросы будут перенаправляться в защищенный контур. Скорей всего gateway будет самописным приложением,
для имплементации кастомной логики маршрутизации запросов на основе гео данных пользователя и близости к нему 
дата-центра с контентом. Также отдельным будет auth сервис.
