#Задание 1

##Выбор фреймворка и стратегия разбития на микросервисы

Для реализации проекта выбран фреймворк Module Federation по нескольким причинам:

- общие зависимости
У нас есть компоненты и библиотеки, которые повторяются в разных частях приложения (например, PopupWithForm, React, CurrentUserContext и т.д.). 
Module Federation позволяет вынести их в «shared» и использовать одну копию вместо дублирования в каждом микрофронтенде. Это упрощает поддержку кода и уменьшает размер бандлов.

- высокая скорость отклика
Мы хотим, чтобы все действия (добавление/удаление карточек, редактирование профиля и т.д.) отражались в интерфейсе максимально быстро.
Module Federation в этой концепции помогает загружать нужные независимые микрофронтенды, которые могут самостоятельно реагировать на события и обновляться, и при этом шарить общий стейт, что дает возможность интерфейсу сразу обновить все компоненты, использующие зависимость (например CurrentUserContext).

- взаимодействие между компонентами
Компоненты разных микрофронтов отображаются на одной странице и должны как-то «общаться». Это можно решать через шаринг состояния (context или store) в Module Federation. 
В будущем при появлении более сложного взаимодействия между микрофронтами можно использовать дополнительно Pus/Sub подход.

- один фреймворк
Мы используем React во всех микрофронтах, поэтому удобно шарить react и react-dom в конфигурации Module Federation. Это избавляет нас от проблемы «двух Реактов» и 
позволяет централизованно управлять версией библиотеки.

Таким образом, Module Federation даёт нам возможность разбить приложение на независимые микрофронты, но при этом легко разделять общие зависимости, экономить на размерах бандлов 
и гибко настраивать взаимодействие между частями приложения.


При проектировании микросервисов были использованы стратегии Вертикальная нарезка. Выделено три логических области, которые практически не пересекаются друг с другом.
Каждый микрофронт получает необходимую информацию из своего апи.

Общение между микрофронтами там, где оно нужно (обновление профиля юзера, получения айди юзера для получения карточек)
реализовано через библиотеку глобального состояния. Так мы сможем быстро обновить информацию о пользователе (CurrentUserContext) 
без задержек и лишних обращений к апи.При этом компоненты достаточно автономны и рендерят сами себя, host-app лишь загружает их через Module Federaion.
В связи с простотой логики сайта и связанностью компонентов (все отображаются на одной странице) механизм Pub/Sub и взаимодействие по апи выглядит менее хорошим вариантом:
при общении по апи мы получаем задержки, а механизм Pub/Sub слишком сложен для данного кейса.

##Микрофронтенды и их компоненты

1. auth-microfrontend (Логин и авторизация)

Login - форма входа пользователя.
Register - форма регистрации.
Header - шапка, где отображается email (при залогиненом пользователе).
InfoTooltip - вспомогательный класс

Все, что касается аутентификации (логин, регистрация, вывод email в Header), удобно держать в одном месте, так как у них общие зависимости 
(в частности auth - апи) и высокая логическая связанность.
Header показывает email, значит, он тоже зависит от состояния авторизации.
При логине микрофронт обновляет переменную loggedIn в CurrentUserContext, который шарится между микрофронтами через Module Federation в host-app. Эта переменная
нужна для ProtectedRoute (см. host-app).

2. cards-microfrontend (Сервис карточек)

CardsPage - основной список карточек (рендер всех карточек, часть бывшего Main.js, относящийся к карточкам)
Card - отдельный компонент карточки (лайки, удаление).
ImagePopup - просмотр увеличенного изображения карточки.
AddPlacePopup - форма добавления новой карточки.
cardsApi - логика работы с карточками на бэке (GET, POST, DELETE, лайки) (часть файла Api.js, относящаяся к карточкам)

Всё, что относится к карточкам (отображение, добавление, лайки, попапы), будет жить в одном микрофронте.

3. profile-microfrontend (Редактирование профиля)

EditProfilePopup - форма изменения имени/описания пользователя.
EditAvatarPopup - форма изменения аватара пользователя.
ProfilePage - рендер профиля (часть бывшего Main.js, относящийся к профилю)
profileApi - получение и обновление информации о пользователе на бэке (часть файла Api.js, относящаяся к профилю)

Логика редактирования профиля, аватара и т.д. отделена от карточек и авторизации, поэтому логично вынести ее в отдельный микрофронт.

4. host-app (Общее приложение)

App.js - хранит верхнеуровневый роутинг (/auth, /cards, /profile) и подключает микрофронты.
Footer - не завязан на логику ни одного микрофронта

Shared (через Module Federation)
ProtectedRoute - логика защищённых роутов (шарится между микрофронтами через ModuleFederation для проверки роутов)
PopupWithForm - универсальный попап, который можно переиспользовать (AddPlacePopup, EditProfilePopup и т.д. на его базе).
CurrentUserContext - контекст пользователя, который шарится между микрофронтендами через Module Federation. В cards-microfrontend нужен id пользователя, в profile-microfrontend - 
контекст для изменения имени и описания юзера. 

host-app управляет основным роутингом (/cards, /profile, /auth), а более детальный роутинг располагается в самих микрофронтендах.
