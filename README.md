# Задание 1. Реализовал первые два уровня

### Уровень 1. Проектирование

Для разбиения монолита фронтенда на микрофронтенды выбрал фреймворк Webpack Module Federation.

Из преимуществ:

- Простота подключения и использования
- Lazy loading
- Shared dependencies
- Большое комьюнити

Еще одна причина в пользу Webpack Module Federation - нет необходимости использовать разные \
JavaScript-фреймворки на одной странице
(что является ключевой особенностью другого фреймворка Single SPA)

В качестве стратегии проектирования выбрал вертикальную нарезку, так как границы доменов четко определены. 

Выделил три предметные области:

- Регистрация/авторизация пользователей
- Управление фотографиями (загрузка, удаление, учет лайков)
- Управление профилем

В качестве метода интеграции выбрал Runtime, так как есть возможность развертывать модули независимо \
и сделать масштабирование более гибким \
(например, масштабировать только модуль фотографий, так как он более популярен и потенциально более нагружен)

### Уровень 2. Планирование изменений

На основании выделенных предметных областей и анализа кодовой базы, разделил монолит на следующие микрофронтенды:

- auth-microfrontend
- profile-microfrontend
- card-microfrontend
- host (точка входа в приложение Mesto)
- shared - шаренные данные между всеми микрофронтендами (изображения, стили, общие компоненты)
